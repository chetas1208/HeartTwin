# HeartTwin Lab — Testing Guide

> **Educational cardiac simulation only. Not for diagnosis or treatment decisions.**

This document describes the HeartTwin Lab test and deployment-validation system:
what it proves, how to run it, and what is intentionally optional.

## Test philosophy

HeartTwin Lab must prove it does *real* work, not fake orchestration:

- The **deterministic cardiac core** (SV, EF, CO, MAP, RR, PV loop, recovery
  trajectories) is computed in Python and golden-tested. LLMs never compute math.
- The **8-agent pipeline** runs end-to-end and is asserted stage by stage.
- **Optional integrations fail safely**: with no OpenAI key, no Weave key, no
  Redis, and VISTA-3D disabled, the pipeline still produces real output via
  deterministic fallbacks — and the system *says so* (it never fakes success).
- **No secrets leak** from public endpoints.
- **No diagnosis/treatment language** appears in user-facing output.

All tests use **synthetic, non-PHI fixtures** (see `fixtures/hearttwin/`). No real
datasets are downloaded or committed.

## Synthetic fixtures

Generated by `scripts/create_synthetic_fixtures.py` (idempotent, offline):

| Fixture | Use |
|---|---|
| `manual_baseline.json` | Baseline vitals + expected SV 70 / EF 58.33 / CO 5.04 / MAP 93.33 / RR 833.33 |
| `manual_reduced_function.json` | Reduced pump function input (SV 55 / EF 36.67 / CO 4.84 / MAP 101.67 / RR 681.82) |
| `manual_partial_data.json` | Partial data (EDV/ESV missing) — must warn, not invent SV |
| `report_*.txt` | Synthetic report text for extraction |
| `ecg_synthetic_normal.csv`, `ecg_synthetic_fast.csv` | Synthetic ECG-like waveforms |
| `echo_metadata_*.json` | Synthetic echo metadata (no video) |
| `vista3d_success_response.json`, `vista3d_failure_response.json` | VISTA-3D client parsing/fail-safe |

These are inspired only by the *structure* of public datasets (see
[`docs/datasets.md`](./datasets.md)): EchoNet-Dynamic, PTB-XL, CAMUS, ACDC,
MIMIC-IV-ECG, MIMIC-IV-Note, VISTA-3D, TotalSegmentator. **No real records are
used.**

## Running tests

### Backend unit/integration (Python)

```bash
pnpm test:py            # python -m pytest python/hearttwin/tests
```

Key suites:

- `test_env_config.py` — env var spec, deploy modes, config endpoint has no secrets
- `test_cardiac_formulas_golden.py` — golden SV/EF/CO/MAP/RR + impossible-state rejection
- `test_pipeline_integration.py` — full pipeline on baseline / reduced / partial fixtures
- `test_openai_fallbacks.py` — every agent works with `OPENAI_API_KEY` removed
- `test_weave_integration.py` — local trace fallback, no secret/PII logging
- `test_redis_memory.py` — in-memory fallback, JSON-only, no raw report storage
- `test_vista3d_client.py` — disabled/enabled/failure/timeout handling
- `test_safety_language.py` — unsafe-phrase scan + required-phrase presence
- `test_api_routes.py` — all routes via FastAPI TestClient
- `test_model_routing.py`, `test_agent_schemas.py` — model env routing + shared schema

### API tests (Playwright request context — no browser rendering)

```bash
pnpm test:api           # playwright test tests/api
```

These hit a running API. Point them at a server with `E2E_API_BASE` (or
`E2E_BASE_URL`, which derives `<url>/api/v1`). Example against a local uvicorn:

```bash
python -m uvicorn python.hearttwin.api:app --port 8077 &
E2E_BASE_URL=http://127.0.0.1:8077 E2E_API_BASE=http://127.0.0.1:8077/api/v1 pnpm test:api
```

### E2E browser tests (Playwright + Chromium)

```bash
pnpm exec playwright install chromium   # once
pnpm test:e2e                           # playwright test tests/e2e
```

By default `playwright.config.ts` starts a local dev server (`pnpm dev:full`,
frontend + Python API). To test an already-running or deployed instance:

```bash
E2E_BASE_URL=https://your-app.vercel.app pnpm test:e2e
```

Specs:

- `00-health-and-config` — health, config safety, system-check (request only)
- `01-root-routing` — `/`, `/lab`, `/about`, safety phrase, CTA
- `02-lab-manual-baseline` — UI: enter baseline vitals, operate, see real metrics
- `03-full-pipeline-baseline` — full pipeline, 8 traced stages, no unsafe language
- `04-partial-data-safety` — warnings, labeled priors, no invented SV
- `05-harness-weave-redis-vista` — harness statuses + self-improve gating
- `06-case-page-refresh` — `/case/{id}` survives reload
- `07-production-smoke` — spine smoke (request + page)

### Production smoke

```bash
E2E_BASE_URL=https://your-app.vercel.app pnpm smoke:prod
# or in-process, no server:
python scripts/run_local_smoke.py
```

### Verification scripts

```bash
pnpm verify:env      # env var spec, .env.example has no real secrets
pnpm verify:repo     # monorepo structure, 8 agents, tools, docs, fixtures
pnpm verify:vercel   # root deploy readiness, vercel.json, no hardcoded localhost
pnpm verify:all      # all of the above + check + build + test:py + test:api
```

## Enabling external integration tests

Tests that would call real OpenAI / Weave / Redis / VISTA endpoints are
**skipped by default**. Enable them explicitly:

```bash
RUN_EXTERNAL_INTEGRATION_TESTS=true pnpm test:py
```

You must also provide the relevant credentials in the environment. Without the
flag, those tests are skipped (and reported as skipped, never as passed).

## What each env var controls

See [`docs/deployment-vercel.md`](./deployment-vercel.md) for the full table.
Summary:

- **OpenAI** (`OPENAI_API_KEY`, `OPENAI_MODEL_*`) — agent reasoning; missing key →
  deterministic fallbacks. Models are routed per agent via env, never hardcoded.
- **Weave** (`WANDB_*`, `NUXT_PUBLIC_WEAVE_PROJECT_URL`) — tracing; missing key →
  local trace fallback.
- **Redis** (`UPSTASH_REDIS_REST_*`, `HEARTTWIN_REDIS_MEMORY_ENABLED`) — case/agent
  memory; missing → in-memory fallback.
- **Blob** (`BLOB_READ_WRITE_TOKEN`) — file storage; missing → local metadata fallback.
- **API base** (`NUXT_PUBLIC_API_BASE`, `API_BASE`, `NEXT_PUBLIC_API_BASE` fallback).
- **VISTA-3D** (`VISTA3D_*`) — optional 3D segmentation; disabled/failsafe by default.
- **App** (`HEARTTWIN_SAFETY_MODE=strict`, `HEARTTWIN_TRACE_MODE`,
  `NUXT_PUBLIC_APP_NAME`).

## Known limitations

- **Browser E2E requires system libraries.** Headless Chromium needs OS packages
  (e.g. `libasound2`). On a fresh CI/host run `pnpm exec playwright install --with-deps chromium`
  (needs root). In restricted sandboxes without those libs, the Node-based
  **API/request specs still run**, but browser-rendering specs cannot launch.
- External integration tests are opt-in (see above) and require real credentials.
- The deterministic golden values are fixed; changing a formula requires updating
  the golden tests *and* must be justified by a proven bug.
